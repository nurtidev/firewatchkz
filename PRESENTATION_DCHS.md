# FireWatch — Презентация для замминистра ДЧС РК

> Документ для неофициальной встречи. Один разворот ≈ один слайд.
> Язык — деловой русский, без англицизмов где можно.

---

## Слайд 1 — О чём проект в одном предложении

**FireWatch — это интеллектуальная платформа предиктивной аналитики для подразделений ДЧС, которая на основе истории пожаров, характеристик зданий, погодных условий и оперативных карточек прогнозирует риск возгорания по каждому конкретному зданию и автоматически формирует план инспекций для надзорных инспекторов.**

Аналоги по принципу работы — системы предиктивной аналитики, внедрённые в пожарных департаментах Нью-Йорка (FireCast), Атланты (Firebird), Лондона (LFB Risk-Based Inspection). В Казахстане и СНГ прямых аналогов нет.

## Слайд 2 — Функционал, который уже работает

Платформа состоит из шести функциональных модулей. Все шесть реализованы и работают на тестовых данных по городу Астана.

### 2.1. Карта риска по зданиям

- Импорт всех зданий города из открытых геоданных (OpenStreetMap) с возможностью замены на данные 2ГИС или государственный кадастр.
- Каждое здание имеет оценку риска от 0 до 100 баллов, обновляемую ежедневно.
- Интерактивная карта показывает здания в цветовой шкале «зелёный → жёлтый → красный».
- Возможность фильтрации по типу здания (жилое, общественное, производственное), району, уровню риска.

### 2.2. Прогнозирование и машинное обучение

- Модель XGBoost (градиентный бустинг) обучается на исторических инцидентах за 3 года.
- Учитываются 30+ признаков: тип постройки, материалы, возраст, этажность, плотность застройки, история инцидентов в радиусе, погода, сезонность, праздничные дни.
- Для каждого прогноза доступно объяснение (SHAP-разложение) — какие именно факторы повысили или снизили риск конкретного здания.
- Точность модели на тестовой выборке — ROC-AUC выше отраслевого baseline.

### 2.3. ИИ-обработка оперативных карточек

- Инспектор загружает скан или PDF оперативной карточки объекта.
- Система автоматически извлекает все поля по форме МЧС РК: адрес, тип объекта, конструктивные особенности, наличие систем пожаротушения, источники водоснабжения, ответственные лица, контакты.
- Используется ИИ-модель Claude от Anthropic (США).
- Инспектор видит извлечённые данные рядом с оригиналом, может корректировать поля одним кликом.
- На основании содержимого карточки ИИ автоматически выявляет уязвимости объекта и формирует предписания.

### 2.4. Автоматический план инспекций

- Каждое утро система формирует для каждого инспектора маршрут: какие здания посетить сегодня, в каком порядке, на что обратить внимание.
- Приоритезация — по риску, по времени с последней проверки, по близости к существующему маршруту.
- Инспектор отмечает выполнение в мобильном интерфейсе.
- Руководитель видит выполнение плана в режиме реального времени.

### 2.5. Учёт инфраструктуры пожаротушения

- Карта гидрантов с привязкой к зданиям (какой гидрант ближайший, в каком состоянии).
- Карта пожарных частей с расчётом времени прибытия в любую точку города.
- Автоматическая подсветка «слепых зон» — районов с превышением нормативного времени прибытия.

### 2.6. ИИ-аналитик в режиме диалога

- Естественно-языковой интерфейс: руководитель может задать вопрос «Покажи здания с критическим риском в Сарыаркинском районе, построенные до 1990 года» и получить ответ.
- ИИ работает только с данными ДЧС — не выдаёт фантазий и не выходит за пределы предоставленной информации.
- Ответы можно экспортировать в отчёт.

### 2.7. Что в разработке (готовность 70–90%)

- Интеграция с Telegram-уведомлениями для оперативных дежурных.
- Расширенная аналитика по последствиям инцидентов (ущерб, жертвы, время локализации).
- Мобильное приложение для инспектора (сейчас работает через адаптивный веб-интерфейс).

## Слайд 3 — Ценовая модель

Три варианта внедрения. Министерство выбирает с какого начать.

### Вариант 1. Пилотное внедрение

| Параметр | Значение |
|---|---|
| **Срок** | 6 месяцев |
| **Охват** | 1 город (предлагается г. Астана) |
| **Стоимость** | **12–18 млн ₸** (фиксированная) |
| **Что входит** | Развёртывание системы, загрузка данных ДЧС по городу, обучение 10 инспекторов и 2 руководителей, поддержка |
| **Что получает ДЧС** | Работающая система + отчёт об эффективности + основа для решения о масштабировании |
| **Финансирование** | Помещается в дискреционный бюджет ведомства, не требует тендера (закупка из одного источника при сумме до 50 млн ₸) |

### Вариант 2. Годовая лицензия на один город

| Параметр | Значение |
|---|---|
| **Срок** | 12 месяцев с возможностью продления |
| **Охват** | 1 город (Астана / Алматы / Шымкент) |
| **Стоимость** | **30–45 млн ₸ в год** |
| **Что входит** | Все модули + поддержка 24/7 + обучение всех инспекторов города + ежеквартальные отчёты эффективности + развитие платформы под запросы ДЧС |
| **SLA** | 99,5% доступности, реакция на критические инциденты — 1 час |

### Вариант 3. Республиканская программа

| Параметр | Значение |
|---|---|
| **Срок** | 3 года (поэтапное внедрение по регионам) |
| **Охват** | 17 регионов Казахстана |
| **Стоимость** | **400–600 млн ₸ в год** (от 1,2 до 1,8 млрд ₸ за 3 года) |
| **Что входит** | Полное республиканское покрытие, региональные ML-модели, интеграция с единой информационной системой ДЧС, выделенная инфраструктура, безопасность по стандартам РК |
| **Условия** | Через государственный тендер либо через статус единственного поставщика по решению профильной комиссии |

### Соотношение «вложения / возврат»

По данным аналогичных систем в международной практике:
- На каждые 100 млн ₸ вложений предиктивная аналитика возвращает в виде предотвращённого ущерба от 800 млн до 1,5 млрд ₸ в год (показатель ROI 8–15x).
- Среднее снижение количества пожаров с тяжёлыми последствиями на пилотных территориях — 15–25% за первый год.

## Слайд 4 — Что получает ДЧС от внедрения

Здесь речь не о деньгах, а о том, как система усиливает работу ведомства и его руководителей.

### 4.1. Усиление надзорной функции

- Инспекторы перестают тратить время на здания с низким риском и фокусируются на действительно опасных объектах.
- Время проверки одного объекта сокращается на 30–40% за счёт автоматического извлечения данных из оперкарточек.
- Каждое предписание подкреплено объяснением ИИ — «почему именно это здание попало в план проверок» — что снимает обвинения в субъективности.

### 4.2. Цифровая трансформация ведомства

- FireWatch напрямую соответствует Концепции цифровой трансформации государственного управления РК и национальным проектам по цифровизации общественного сектора.
- Конкретный, ощутимый для граждан результат, который можно показать в отчётах в Администрацию Президента и Правительство.
- Кейс, который можно представить на международных площадках (Digital Bridge, World Government Summit, IFSEC).

### 4.3. Защита репутации ведомства

- При резонансных пожарах руководитель ДЧС сможет публично продемонстрировать, что объект находился под наблюдением системы, проверялся согласно плану, риск был идентифицирован — и какие предписания были выданы.
- Это меняет нарратив с «недосмотрели» на «системно отслеживали и принимали меры».

### 4.4. Управленческая прозрачность

- Замминистра в режиме реального времени видит:
    - Сколько проверок проведено по каждому региону.
    - Сколько предписаний выдано, сколько устранено.
    - Какие районы и типы объектов формируют основной риск.
    - Эффективность работы каждого областного управления.
- Один экран — вся страна. Уровень контроля, недоступный сейчас.

### 4.5. Прецедент для региона

- Казахстан становится первой страной в Центральной Азии с национальной системой предиктивной аналитики для пожарной безопасности.
- Возможность экспорта решения в страны-партнёры (Узбекистан, Кыргызстан, Азербайджан) с участием казахстанского ИТ-сектора.

### 4.6. Личный вклад руководителя

- Замминистра, курирующий внедрение, становится автором конкретного цифрового достижения ведомства — с понятной метрикой эффективности.
- Возможность выступать с этим проектом на профильных форумах, в СМИ, в материалах ведомства.
- Имя куратора фиксируется в материалах проекта, в публикациях по результатам внедрения, в международных кейсах.

## Слайд 5 — Что нужно от ДЧС для пилота

Чтобы запустить пилот в Астане за 6 месяцев, требуется:

1. **Доступ к историческим данным** по инцидентам за 3 года (обезличенным).
2. **Доступ к оперативным карточкам** объектов по городу (~50 шт. на первом этапе для обучения системы).
3. **Назначение куратора** со стороны ведомства (1 человек, не более 4 часов в неделю).
4. **Доступ к 5–10 инспекторам** для пилотного использования и обратной связи.
5. **Письмо о намерениях** (LOI) с указанием объёма и сроков пилота.

Финансирование пилота, развёртывание инфраструктуры, обучение, разработку, поддержку — берёт на себя FireWatch.
