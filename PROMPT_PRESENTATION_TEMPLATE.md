# Универсальный промпт-шаблон для презентации проекта

> Используется для генерации презентации любого проекта под любую аудиторию:
> инвестор, госзаказчик, корпоративный клиент, партнёр, внутренний стейкхолдер.
>
> Промпт работает с любым LLM-агентом (Claude Code, Codex, ChatGPT, Cursor).
> Перед использованием заполни блок `<<КОНТЕКСТ>>` своими данными.

---

## Промпт (копировать всё ниже до закрывающей метки)

````
Ты готовишь презентацию проекта для встречи с конкретной аудиторией.
Документ — markdown-файл, где один раздел = один слайд (6–9 слайдов + приложения).

═══════════════════════════════════════════════════════════════════

═══════════════════════════════════════════════════════════════════

1. ПЕРВЫМ ДЕЛОМ прочитай актуальное состояние кода проекта в этой
   директории. Не выдумывай функционал, описывай только то, что
   реально реализовано. Проверь README, TASKS / TODO, структуру
   директорий, migrations, основные роуты / страницы.
2. Если в коде функционала меньше чем заявлено в <<КОНТЕКСТ>>, или
   меньше 3 рабочих модулей — скажи об этом первым делом и спроси
   что делать (описывать «в разработке», отложить, или продолжать
   с тем что есть).
3. Не приписывай интеграции, сертификации, юридические статусы,
   партнёрства, метрики, которые не подтверждены.
4. Цены и денежные числа давай диапазоном, не одной цифрой, если
   аудитория — клиент или госзаказчик (одна цифра становится потолком
   в переговорах). Для инвестора — точная сумма раунда обязательна.
5. Тон и язык — как указано в <<КОНТЕКСТ>>. По умолчанию деловой,
   без избыточных англицизмов.
6. Не пиши маркетинговых эпитетов («революционный», «уникальный»,
   «не имеющий аналогов»). Заменяй конкретикой.

СТРУКТУРА ДОКУМЕНТА — выбери набор слайдов по типу аудитории

ОБЩИЕ СЛАЙДЫ (всегда):
  S1. Один абзац: что делает проект (одно жирное предложение +
      контекст). Аналоги если есть.
  S2. Функционал / продукт — что реально работает (модули,
      каждый с 3–5 буллетами).
  S6. Команда и текущий статус (одна команда / один основатель,
      инфраструктура, стадия готовности).

СПЕЦИФИЧНЫЕ СЛАЙДЫ — добавлять по типу аудитории:

▶ Если аудитория [A] ИНВЕСТОР:
  S_problem. Проблема: чья, насколько большая, как сейчас решается.
  S_market. Рынок: TAM / SAM / SOM с источниками цифр. Если оценок
            нет — честно «по нашим оценкам, X».
  S_traction. Трекшн: пользователи, выручка, рост, LOI, пилоты.
              Без трекшена — обозначить «pre-traction, ищем X
              для первых N клиентов».
  S_business. Бизнес-модель: как зарабатываем, юнит-экономика
              (LTV, CAC, AOV если применимо).
  S_competition. Конкуренты: 3–5 игроков, чем мы отличаемся.
                 Не «у нас нет конкурентов» — всегда есть альтернативы.
  S_ask. Раунд: ТОЧНАЯ сумма, оценка (или SAFE / convertible),
         use of funds разбит на 3–5 статей, runway в месяцах,
         следующая контрольная точка.
  S_exit. Долгосрочная история: что построим за 5 лет, варианты
          ликвидности.

▶ Если аудитория [B] ГОСЗАКАЗЧИК (министерство / ведомство):
  S_pricing. Ценовая модель в ТРИ tier-а в виде таблиц:
             - Пилот: укладывается в порог закупки из одного
               источника без тендера для соответствующей страны
               (для РК — до 50 млн ₸ по 67-ФЗ).
             - Годовая лицензия: один регион / город.
             - Республиканская / национальная программа: через
               тендер либо статус единственного поставщика.
             Колонки: срок / охват / стоимость / что входит /
             механизм финансирования.
  S_benefits. Что получает ведомство — 5–7 НЕФИНАНСОВЫХ выгод:
              усиление основной функции, цифровая трансформация
              (связь с национальной повесткой), защита репутации,
              управленческая прозрачность, региональный прецедент,
              личный вклад куратора в легальных форматах
              (упоминание в материалах, наблюдательный совет).
  S_request. Что нужно от ведомства для пилота — 4–6 конкретных
             пунктов (доступ к данным, куратор, инспекторы, LOI).
             Завершить фразой «финансирование / разработку / поддержку
             берёт на себя проект» — снимает страх обязательств.

▶ Если аудитория [C] КОРПОРАТИВНЫЙ КЛИЕНТ B2B:
  S_pain. Проблема клиента: конкретная боль в их операциях, в их
          числах. Желательно с цифрами их индустрии.
  S_roi. ROI: как окупится — экономия часов, снижение издержек,
         рост выручки. Расчёт под их масштаб.
  S_security. Безопасность и compliance: где данные, шифрование,
              сертификации, audit trail, on-premise опция если
              нужна.
  S_integration. Интеграция: как стыкуется с их стеком (SSO, API,
                 SAP / 1C / SFDC / и т.п.).
  S_pricing. Цены — диапазон, 3 tier (пилот / single team /
             enterprise license).
  S_references. Референсы или похожие кейсы (если нет — обозначить
                «вы будете в числе первых, что даёт X преимуществ»).

▶ Если аудитория [D] ПАРТНЁР / ИНТЕГРАТОР:
  S_fit. Чем мы дополняем друг друга: что у нас есть, что у них.
  S_revenue. Модель совместного заработка: revenue share, комиссия,
             white-label, со-брендинг.
  S_what_we_provide. Что мы даём партнёру: продукт, поддержку,
                     обучение, лиды, маркетинг.
  S_what_we_need. Что нам нужно от партнёра: каналы продаж,
                  внедрение, локализация, отношения с клиентами.

▶ Если аудитория [E] ВНУТРЕННИЙ СТЕЙКХОЛДЕР:
  S_alignment. Связь с приоритетами компании / стратегией.
  S_resources. Что нужно: люди, бюджет, время.
  S_milestones. Контрольные точки и метрики успеха.
  S_risks. Риски и митигация — честно, без приукрашивания.

ВСЕГДА В КОНЦЕ:
  S_next. Следующий шаг — 2–3 варианта от самого лёгкого
          (звонок / демо) к самому серьёзному (подписание /
          бюджет / term sheet). Собеседник всегда должен выбрать
          какой-то один — встреча не заканчивается ничем.
  S_appendix. Приложения по запросу: техописание, скриншоты,
              видеодемо, due diligence pack, юридическая
              проработка.

═══════════════════════════════════════════════════════════════════
ФОРМАТ ВЫХОДА
═══════════════════════════════════════════════════════════════════

Файл: PRESENTATION_[АУДИТОРИЯ_ИЛИ_ИМЯ].md в корне проекта.
Заголовок документа: «[Название проекта] — Презентация для [аудитория]».
Каждый слайд начинается с «## Слайд N — [тема]».
Между слайдами — горизонтальная линия (---).

После генерации основной презентации спроси у меня:
1. Сделать ли отдельный документ Q&A — заготовки ответов на ~20–30

   (технические, бизнес, команда, цена, неудобные вопросы,
   контр-вопросы которые задаю сам).
2. Сделать ли one-pager на 1 страницу A4 для распечатки / leave-behind.

Не делай эти документы автоматически — спроси сначала.
````

## Дополнительный промпт — Q&A и one-pager

Запустить после генерации основной презентации:

````
На основе только что созданной презентации сделай два дополнительных файла:

ФАЙЛ 1: PRESENTATION_[...]_QA.md — заготовки ответов на ~25 вопросов,
которые задаст собеседник. Сгруппируй по 7–9 блокам, релевантным типу
аудитории. Для любой аудитории включи:
  • Технические / продуктовые вопросы.
  • Команда и устойчивость («что если основатель пропадёт»).
  • Конкуренция и альтернативы.
  • Цена / стоимость / бюджет.
  • Процесс и сроки.
  • Неудобные вопросы — самый важный блок. Для инвестора это «почему
    не подняли больше», «почему оценка такая», «что если фаундер уйдёт».
    Для госзаказчика — «почему через знакомых», «что хотите лично»,
    «готовы учесть интересы» — ответы должны легально переводить
    разговор в прозрачные форматы и никогда не предлагать теневых
    схем. Для корпоративного клиента — «почему не сделать своими
    силами», «почему доверять стартапу».
  • Контр-вопросы — что я задаю собеседнику когда разговор затихает.

В конце два чек-листа:
  • Что взять на встречу (распечатки, ноутбук, демо, USB, шаблон LOI / term sheet).
  • Что НЕ делать (не извиняться, не обещать невозможное, не торговаться
    с порога, не уходить без следующего шага).

ФАЙЛ 2: PRESENTATION_[...]_ONEPAGER.md — одна страница A4:
  • Название + одно предложение что делает.
  • Компактная таблица функциональных модулей (модуль / что даёт).
  • Компактная таблица цен / условий (если есть).
  • 5–7 буллетов «что получает собеседник» — адаптировано под аудиторию.
  • 4–6 буллетов «что нужно от собеседника» (если применимо).
  • 2–3 варианта следующего шага.
  • Контакт.

Тот же тон и язык что в основной презентации.
````

## Что критично НЕ менять в шаблоне

| Принцип | Зачем |
|---|---|
| «Сначала прочитай код, не выдумывай функционал» | Без этого LLM допишет фичи которых нет — провал на демо или DD |
| Цены диапазоном для клиентов и госзаказчиков | Фиксированная цифра становится потолком; для инвестора — наоборот, точная сумма обязательна |
| Раздельные структуры по типу аудитории | Универсальная структура «для всех» одинаково плохо работает на всех |
| Нефинансовые выгоды для госзаказчика | Сердце B2G питча — без этого получается коммерческая презентация которая не работает на чиновника |
| Блок «неудобные вопросы» в Q&A | Без подготовки встреча может пойти в опасную сторону, особенно с госзаказчиком |
| Следующий шаг как обязательный последний слайд | Встреча без выбранного следующего шага = встреча без результата |
| Запрос на Q&A и one-pager после основного документа, а не автоматически | Иначе LLM генерит три файла когда нужен один — теряется фокус |

## Примеры заполнения <<КОНТЕКСТ>>

**Пример A — питч инвестору:**
```
Название: SaaS для управления складом малого бизнеса
Одно предложение: помогаем малому ритейлу вести учёт остатков без 1С
Стадия: MVP, 50 платящих клиентов, MRR $4K
Аудитория: [A] Инвестор (бизнес-ангел)
Собеседник: Игорь, ангел-инвестор из Алматы
Цель: получить $100K на seed
Язык: русский, тон деловой
Просим: $100K за 10%, use of funds — найм продактов и маркетинг
```

**Пример B — питч в министерство здравоохранения:**
```
Название: AI-помощник для участковых врачей
Аудитория: [B] Госзаказчик (Минздрав)
Собеседник: замминистра по цифровизации
Цель: LOI на пилот в одной поликлинике Астаны
Просим: пилот 6 мес, доступ к 5 врачам и обезличенным данным
```

**Пример C — питч в Kaspi для интеграции:**
```
Аудитория: [C] Корпоративный клиент B2B
Собеседник: руководитель продукта Kaspi Pay
Цель: пилот интеграции на 3 месяца
```
