## Product Vision

FireWatch is a **multi-city B2G SaaS platform** that gives fire safety departments AI-powered predictive analytics.  
The core loop: ingest historical fire incident data → forecast future risk → surface AI recommendations → alert dispatchers.

It is **not** tied to a single city. Cities are first-class configuration — a city selector in the UI lets operators switch contexts. Each city has its own:
- Incident data (CSV or API feed)
- GeoJSON district boundaries
- Fire station locations

**Initial test city: Astana, Kazakhstan** (used for all dev/demo data).  
Target market: municipal governments and fire departments across Kazakhstan, then Central Asia.

---

## Reference Implementation

This project is architecturally modelled on [hse-ai-dashboard](https://github.com/nurtidev/hse-ai-dashboard) — an AI safety dashboard built for KMG-Kumkol (oil & gas HSE).  
Key patterns to reuse:
- FastAPI router structure (`routers/` per domain)
- Holt-Winters ETS forecasting via `statsmodels`
- Claude API integration pattern (context injection + caching)
- Telegram alert system with 4-tier severity
- Railway two-service deployment (`frontend` + `api`)

## Data Strategy

- **Demo / MVP**: synthetic data generated by `backend/scripts/generate_data.py`
  - ~3 years of daily fire incidents
  - Fields: `date`, `city`, `district`, `building_type`, `cause`, `severity`, `casualties`, `damage_tenge`
  - Realistic seasonality: peaks in Jan–Feb (heating) and May–Jun (dry/wind)
- **Production**: OpenData Kazakhstan — https://data.egov.kz (ДЧС datasets)

The data loader (`backend/services/data_loader.py`) must accept both sources transparently.

## Domain Concepts

| Term | Meaning |
|---|---|
| **Incident** | A recorded fire event with location, cause, severity, damage |
| **District** | Administrative sub-division of a city (район) |
| **Building type** | Residential / Commercial / Industrial / Construction site / Other |
| **Cause** | Electrical fault / Open flame / Arson / Children playing / Other |
| **Severity** | Low / Medium / High / Critical |
| **Risk score** | 0–100 composite score per district (incidents + seasonality + building density) |
| **City config** | JSON object: `{ id, name, center_lat, center_lon, zoom, geojson_path, data_path }` |

## API Surface (planned)

```
GET  /cities                        — list available cities
GET  /incidents?city=&district=     — filtered incident list
GET  /forecast?city=&months=3|6|12  — Holt-Winters prediction
GET  /risk-map?city=                — district risk scores (GeoJSON-ready)
GET  /recommendations?city=         — Claude AI recommendations (cached 1h)
POST /chat                          — AI analyst (streaming)
GET  /kpi?city=                     — KPI summary (prevented damage, etc.)
POST /telegram/test                 — send test alert
GET  /telegram/config               — current alert config
```

## Key Decisions

1. **City as a query param, not subdomain** — simpler for MVP, easy to change later.
2. **Synthetic data first** — unblocks frontend work without waiting for real data scraping.
3. **Claude Haiku** — cost-efficient for frequent recommendation calls; upgrade to Sonnet for chat if needed.
4. **No database for MVP** — CSV files loaded into Pandas DataFrames at startup. Add Postgres later.
5. **Monorepo, two Railway services** — `frontend/` and `backend/` as separate Railway services in one project.
